# The Absence Run — Architecture

## Overview
The Absence Run is a Symfony console application that processes pending employee leave requests for a given run date. The work is split into two stages: a single dispatch command reads every request still awaiting a decision and enqueues each one as its own asynchronous unit of work; a message worker then evaluates each request against the company leave policy, decides to approve or reject it, updates the employee's leave balance, and posts the outcome to an external HR system. The architecture follows a light CQRS layering over Symfony Messenger: a thin console command at the edge validates its input, then dispatches a command through a command-bus facade (`$command->execute()`) to a handler that delegates to a domain service; reads are served by separate query handlers via a query-bus facade (`$query->ask()`). The domain service reports decisions through an HR API client behind an interface, and data moving between layers is carried by DTOs rather than arrays. State lives in a local SQLite database, and an idempotent HTTP contract with the HR system, combined with per-request messaging, is what makes the run safe to repeat or resume after a partial failure.

## Components
The `AbsenceRunCommand` (`app:absence:run`) is the entry point. It parses an optional `--date` run date, wraps it in a `RunInput` DTO validated with Symfony Validator constraints, dispatches a `RunAbsence` command through the command bus, and reports how many requests were enqueued from the returned `DispatchSummary` DTO. The bus facades (`CommandBusInterface`/`QueryBusInterface`, implemented over Messenger by `MessengerCommandBus`/`MessengerQueryBus`) separate state-changing commands from reads. `RunAbsenceHandler` delegates to `LeaveRequestProcessor::processPending`, which pages through the pending set by a `(submittedAt, id)` keyset cursor — fetching ids in oldest-submission-first order via the query bus — and dispatches one `ProcessLeaveRequest` command per request onto the `async` transport. Each queued message is handled by `ProcessLeaveRequestHandler`, which calls `LeaveRequestProcessor::processOne`: it decides the single request, mutates its status and the leave balance, posts the result to the HR API, and flushes. The policy maths live in dedicated services behind interfaces — `WorkingDayCounter` (consumed working days, using `PublicHolidayProvider`) and `EntitlementCalculator` (pro-rata/part-time entitlement and carryover validity) — and each evaluation produces a `Decision` value object. The domain model is three Doctrine entities — `Employee`, `LeaveRequest`, and `LeaveBalance` — supported by the `LeaveType` and `LeaveStatus` enums. Reads go through query handlers (`FindPendingRequestIdsHandler` for the keyset-paginated pending ids, `FindApprovedOverlappingHandler` for overlap/sick-credit checks, and `EmployeeBalanceHandler`), each querying the `EntityManager`/`QueryBuilder` directly, while the write-side balance lookup goes through `LeaveBalanceRepositoryInterface`. Outbound reporting is abstracted by `HrApiClientInterface`, implemented for real by `HrApiClient` over Symfony's HTTP client and by `FakeHrApiClient` in tests. A standalone `mock-hr-api/server.php` stands in for the real HR system during local runs.

## Data Flow
The command validates the run-date input and dispatches `RunAbsence` via the command bus; its handler calls `LeaveRequestProcessor::processPending`. The processor repeatedly asks the query bus for the next page of pending request ids — ordered oldest-submission-first by a `(submittedAt, id)` keyset cursor so the whole set is never held in memory and no row is skipped or duplicated across pages — and for each id dispatches a `ProcessLeaveRequest(requestId, runDate)` message to the `async` transport, returning a `DispatchSummary` count that the command prints. A worker (`bin/console messenger:consume async`) then processes each message independently: `processOne` reloads the request, skips it if it is no longer `PENDING` (re-run safety), computes a decision from the employee's contract, the relevant year's leave balance, and the policy rules, and records the decision on the request via `markDecided`. For approvals it adds the consumed days to the employee's balance. It posts a decision payload — employee id, request id, decision, days, and reason — to the HR API with a per-request idempotency key, stores the returned external reference back on the request, and flushes that request's changes. Unprocessable data (e.g. a missing balance) is logged and left pending without retry; an HR transport failure is rethrown so the worker retries the message later. The HR mock persists each decision to a JSON state file keyed by idempotency key so repeated keys replay the original record rather than creating duplicates.

## External Dependencies
The application is built on the Symfony framework (console, HTTP client, dependency injection, Messenger for the command/query buses and the asynchronous per-request transport, and Validator for input validation) with Doctrine ORM and migrations over a local SQLite file, requiring PHP 8.2+ and Composer. The `async` transport is a Doctrine queue by default (`MESSENGER_TRANSPORT_DSN=doctrine://default`), and is overridden to `sync://` under `when@test` so the suite processes each request inline without a running worker. The leave policy in `docs/LEAVE_POLICY.md` is the functional source of truth, including German statutory rules (BUrlG) and the 2025 public-holiday calendars for the Bavaria and Berlin federal states. The only external integration is the HR system's REST contract — `POST`/`GET /v1/leave-decisions` plus a `POST /v1/_reset` helper — which requires bearer-token authentication and a mandatory `Idempotency-Key` header; its base URL and token are configured in `config/services.yaml`. PHPUnit drives the test suite, which boots the kernel against a fresh SQLite schema per test and substitutes the in-memory fake HR client.
