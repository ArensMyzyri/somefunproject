# The Absence Run — Functionalities

## Overview
The Absence Run is an operator-run batch job that closes out pending leave requests for a pay period. An operator runs a single command for a chosen run date to enqueue the work, and a background worker then works through every request awaiting a decision, applies the company's leave policy to approve or reject each one, keeps each employee's vacation balance up to date, and files the decisions with the HR system. It is designed as a backend job that can be re-run safely after a retry or partial failure, since each request is handled as its own independent unit of work.

## Running the absence run
The operator invokes `php bin/console app:absence:run`, optionally passing `--date=Y-m-d` to set the run date (it defaults to today). The run date matters because it determines whether carried-over leave has already lapsed and how pro-rata entitlement is calculated. The command prints a heading and then enqueues each pending request — in oldest-submission-first order — finishing with either a "no pending requests" notice or a count of how many requests were dispatched for processing. The decisions themselves are made by a worker, which the operator runs with `php bin/console messenger:consume async`; it consumes one queued request at a time, decides it, updates the balance, and files the outcome with HR. The seeded sample period is the 2025 leave year with a run date of 2025-04-15.

## Leave evaluation and decisions
For each pending request the system decides approve or reject and records the outcome, the decision date, and a reason on the request. The intended policy (defined in `docs/LEAVE_POLICY.md`) counts a request in working days — excluding weekends and the public holidays of the employee's federal state, and counting flagged half-days as 0.5 — and compares that against the employee's remaining entitlement. Entitlement derives from the contractual annual allowance, pro-rated for mid-year joiners and leavers and scaled for part-time schedules, plus any still-valid carried-over days. Approvals deplete still-valid carryover first and then the current-year allowance; a request that would exceed the remaining balance is rejected in full rather than partially approved. Leave types are treated differently: vacation consumes the balance, unpaid and special leave do not, and a sick note overlapping an approved vacation credits the overlapping working days back. The system also guards against overlapping approved periods and ignores cancelled requests.

## Balance tracking
Each employee has a per-year leave balance recording carried-over days, the date that carryover expires, and the days already used. When a request is approved the system adds the consumed days to that balance, so subsequent requests in the same run see the reduced remaining entitlement. The balance is the running record of how much vacation an employee has left for the year.

## Reporting decisions to the HR system
Every decision is posted to the external HR API as a JSON payload containing the employee id, request id, decision, days, and reason. Calls authenticate with a bearer token and carry an idempotency key so that re-running the script does not create duplicate records — the HR system replays the original record for a key it has already seen. The identifier the HR system returns is stored back on the request as its external reference. A bundled mock HR server reproduces this contract locally, persisting state to a JSON file and offering a reset endpoint to wipe recorded decisions between runs.

## Configuration and operational modes
The HR API base URL and token are configured in `config/services.yaml`; in a real deployment these would come from the environment or a secrets store. Per-request processing is queued through a Symfony Messenger transport (`MESSENGER_TRANSPORT_DSN`, a Doctrine queue by default), so dispatch and decision-making are decoupled and a worker can be scaled or restarted independently. The application uses a local SQLite database created via Doctrine schema/migration commands and seeded with a realistic sample period through Doctrine fixtures. Behaviour is verified by a PHPUnit suite that runs against a fresh in-memory schema with a fake HR client and the queue switched to synchronous handling, so each request is processed inline and decision logic can be exercised without a live database, worker, or HR endpoint.
